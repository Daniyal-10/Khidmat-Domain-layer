# TD-03 — The Humanitarian Lifecycle (Business View)

## A Note on Method for This Topic (per this turn's explicit instruction)

This dossier prioritizes **discovery of business reality over literature summarization**. External sources below are treated as *evidence* for testing a claim about how humanitarian work actually happens — never as the deliverable itself. Every major Finding is required to answer four questions beyond source/confidence: what it tells us about actual operating practice, why it matters to Khidmat's business understanding, whether it is universal/common/context-specific, and whether it introduces, refines, or challenges an existing internal business concept.

**Terminology note, stated once:** this session's instruction uses the ladder Observation → Inference → Conclusion → Recommendation. This dossier uses Observation, Inference, and Conclusion (the last of these is what the Business Discovery Blueprint calls a **Finding** — tiered, sourced, confidence-rated, per Blueprint §4.1). It does **not** produce Recommendations. The Blueprint's own scope rules (§3, "What does NOT belong") and non-goals (§12) exclude recommending what Khidmat should build — that would be BMP/downstream content, not discovery. ## Tier C Collection (Step 2)

**Observations** (direct textual facts from internal artifacts — evidence of this project's own stated intent, not independent evidence of reality):

- **TC3-1.** `KHIDMAT_BUSINESS_LOGIC_BLUEPRINT_V1.md` §14 states the flow: *Awareness → Lead Creation → Registration → Verification → Needs Assessment → Eligibility/Approval → Support Planning → Volunteer Assignment → Support Delivery → Follow-up → Case Management → Outcome Measurement → Impact Measurement → Knowledge Graph Learning.*
- **TC3-2.** The same section states: *"At every stage a beneficiary may question or correct a decision, and new information... can send the case back to the appropriate earlier stage. The journey is not strictly one-directional."*
- **TC3-3.** The same section states eligibility is *"a human decision realised as a verification outcome plus an approved case plan... not an automatic status flip."*
- **TC3-4.** `GLOSSARY.md` defines **Engagement Stage** as *"the beneficiary's administrative relationship to the Khidmat ecosystem (identified, registered, active, engaged, monitored, exited, etc.)"* and **Human Development Stage** as *"the beneficiary's or household's own position in the humanitarian developmental trajectory — crisis, stabilization, recovery, self-reliance, resilience, or community contribution"* — explicitly stating the two must *"never be conflated."*

**Inference drawn from these observations (labeled as inference, not yet a Finding):** Khidmat's internal artifacts already assume (a) a single, long case-level lifecycle running from first contact through impact measurement and organizational learning, (b) that this lifecycle is loop-capable rather than strictly linear, and (c) that a person's underlying life trajectory is tracked separately from their administrative status in the system. Whether each of these three assumptions matches how humanitarian lifecycles actually function in the wider sector — rather than being Khidmat-specific inventions — is what Tier B/D collection below tests.

## Findings — Conclusions Drawn From Evidence (Step 11)

**BD-TD03-001 — A programme-level (coordination) lifecycle and a case-level (individual) lifecycle are two structurally distinct things, not one.**

*Observation:* The IASC Humanitarian Programme Cycle defines a cyclical, organization/sector-level process — needs assessment & analysis → strategic response planning → resource mobilization → implementation & monitoring → operational review & evaluation (the 2026 HPC discussion paper simplifies this toward four elements: needs analysis, response planning, financial tracking, response monitoring). This cycle governs how the *humanitarian system* plans and resources itself, typically on an annual or crisis-length cadence — it is not a description of what happens to one specific person. *Source:* Tier B, IASC/UNHCR/IOM/WHO Health Cluster (4 independent institutional sources). *Confidence: High.*

- **What this tells us about actual operating practice:** Humanitarian organizations run at least two structurally distinct lifecycle rhythms concurrently — a slower, collective planning rhythm that decides what capacity and resources exist, and a faster, individual, event-driven rhythm that governs how one person's case unfolds within whatever capacity the first rhythm made available. These are coupled but not the same clock, and conflating them would misdescribe how either actually works.
- **Why this matters for Khidmat's business understanding:** BMP Chapter 3 asks for "the business-level lifecycle... how a humanitarian case unfolds." Read literally, this scopes Chapter 3 to the case-level rhythm — consistent with Khidmat's own `KHIDMAT_BUSINESS_LOGIC_BLUEPRINT_V1.md` §14, which is case-level throughout. This Finding's value is making that scoping decision *explicit* rather than implicit: Chapter 3 should state plainly that it describes the case-level lifecycle and that a second, related, programme-level cycle exists and constrains it (e.g., resource mobilization outcomes bound what Support Planning/Delivery can actually offer a given case) — without attempting to describe that second cycle itself, which is not this topic's evidence base.
- **Universal, common, or context-specific:** Common, near-universal practice specifically within UN-coordinated humanitarian response (the HPC is IASC's own standard mechanism). Not yet checked against smaller, non-UN-coordinated, or purely community/faith-based response — an open gap, not assumed to generalize further than the evidence supports.
- **Introduces, refines, or challenges an existing concept:** **Refines** — it does not contradict Khidmat's case-level lifecycle, but surfaces a scope boundary that was previously implicit rather than stated, and names a second, related lifecycle concept (programme/resourcing cycle) that this project has not yet named anywhere internally.

---

**BD-TD03-002 — Real case lifecycles are structurally non-linear; this directly and strongly corroborates Khidmat's own "accountability loop" principle.**

*Observation:* Two independent, authoritative interagency case-management standards (GBV Case Management Guidelines; Child Protection Case Management Guidelines) and supporting case-management literature describe a six-stage cycle (screening/intake, assessment, planning, implementation, monitoring, closure) explicitly as forming *"a loop rather than a straight line — monitoring routinely sends a case back into reassessment."* *Source:* Tier B, two interagency standard-setting bodies + Tier D corroboration (Better Care Network, Sopact). *Confidence: High.* Corroboration: ≥4 independent source families.

- **What this tells us about actual operating practice:** Reopening a case, or sending it back to an earlier stage as circumstances change, is standard professional practice — not an exception path bolted on for edge cases. A lifecycle description that models only forward progression would misdescribe how case work actually happens.
- **Why this matters for Khidmat's business understanding:** TC3-2 already asserts this internally ("the journey is not strictly one-directional"). This Finding's value is confirming that assertion is a match to documented sector standard practice, not a Khidmat invention requiring independent justification — Chapter 3's author can write the lifecycle narrative as loop-capable with external grounding, not merely internal assertion.
- **Universal, common, or context-specific:** Common/standard practice, corroborated across two distinct case-management sub-domains (GBV, child protection); not yet confirmed for every humanitarian domain (e.g., a one-time cash distribution "case" may not loop the same way a protection case does) — stated as a scope limit, not assumed away.
- **Introduces, refines, or challenges an existing concept:** **Refines and strengthens** — externally validates the existing internal accountability-loop concept rather than contradicting it.

---

**BD-TD03-003 — Sector-standard case lifecycles end at closure; Khidmat's internal lifecycle continues into outcome, impact, and organizational-learning stages not present in the base sector model — this is a genuinely open question, not a settled fact either way.**

*Observation:* The interagency case-management stage sets found (screening/intake → assessment → planning → implementation → monitoring → closure) end at closure. Khidmat's internal `KHIDMAT_BUSINESS_LOGIC_BLUEPRINT_V1.md` §14 continues past an equivalent closure point into Outcome Measurement, Impact Measurement, and Knowledge Graph Learning, as further stages of the *same* named lifecycle. *Source:* Tier B (case-management guidelines, above) compared directly against Tier C (BLB §14). *Confidence: Medium* — the textual comparison itself is high-confidence; whether treating outcome/impact/learning as literally further stages of one case lifecycle (rather than a related-but-separate discipline) is itself sector-recognized practice was not independently corroborated by any source describing them as unified.

- **What this tells us about actual operating practice:** The sector sources found treat "did the case close" (a case-management milestone) as related to, but organizationally distinct from, "did it work" (typically framed in the literature as a separate monitoring-and-evaluation function, often on a different cadence, sometimes performed by a different team).
- **Why this matters for Khidmat's business understanding:** This is a genuine, unresolved design question for BMP Chapter 3, surfaced by discovery rather than assumed. Chapter 3's own "Expected output" (BMP Blueprint §6) explicitly requires describing the lifecycle "independently of any future technical engagement-stage taxonomy" — this Finding shows that Khidmat's *existing* internal artifact (BLB §14) may itself already be the kind of implementation-influenced structure BMP Blueprint §7's Q2 warned about, at exactly this seam. Whether Chapter 3 should treat outcome/impact/learning as part of the same business-level lifecycle narrative, or as a related-but-distinct business capability that *consumes* the lifecycle's output, should be a deliberate authorial decision, not something silently inherited from BLB §14.
- **Universal, common, or context-specific:** The base six-stage cycle through closure reads as common/near-universal in the sources found. The "M&E as a separate discipline" framing is a common pattern in what was reviewed, but was not exhaustively tested against models that do formally include outcome tracking inside the case lifecycle — held at Medium confidence for this reason, not asserted as settled.
- **Introduces, refines, or challenges an existing concept:** **Challenges** an assumption Khidmat's own internal artifact currently makes implicitly (that outcome/impact/learning are simply further stages of one case lifecycle) by showing sector practice more commonly separates them — flagged here as a decision point for BMP Chapter 3's author, not resolved by this dossier.

---

**BD-TD03-004 — A person's underlying developmental trajectory is tracked separately from programme-enrollment status in comparable, at-scale sector practice — corroborating Khidmat's Engagement-Stage / Human-Development-Stage split.**

*Observation:* Graduation Approach literature — a livelihoods/poverty-programming model originated by BRAC, now used at scale including by USAID's Bureau for Humanitarian Assistance specifically for displaced-population programming — explicitly treats "graduation" (an approach-specific completion milestone) as distinct from a household's ongoing eligibility/status in other programs; these are documented as two separately tracked concepts in mature practice, not one combined status field. *Source:* Tier D, ≥4 independent source families (an academic policy lab, an implementing-NGO field guide, a funder evaluation, and a peer-reviewed systematic review specific to displaced populations). *Confidence: Medium-High.*

- **What this tells us about actual operating practice:** At-scale humanitarian/development livelihoods programming already separates "where is this household in its underlying life/poverty trajectory" from "what is this household's status in this specific program" — this distinction is not run as a single combined field in mature, funded, evaluated practice.
- **Why this matters for Khidmat's business understanding:** This directly corroborates `GLOSSARY.md`'s existing internal distinction between Human Development Stage and Engagement Stage, which that document already insists must "never be conflated." This Finding shows that instruction is not an over-engineered internal abstraction but a pattern already proven necessary in comparable, real, evaluated humanitarian programming — specifically including displaced-population contexts, which is directly relevant to Khidmat's own likely applicability.
- **Universal, common, or context-specific:** Common practice specifically within livelihoods/poverty-graduation programming, including explicitly in displaced-population contexts (the PMC systematic review) — narrower than "all humanitarian work broadly," but squarely on-point for the household-resilience/recovery-trajectory concepts Khidmat's own artifacts already center.
- **Introduces, refines, or challenges an existing concept:** **Refines and strengthens** — externally validates a distinction the project had already drawn internally but had not previously checked against sector practice.
