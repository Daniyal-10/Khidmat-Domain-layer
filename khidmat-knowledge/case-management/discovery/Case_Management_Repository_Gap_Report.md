# Case Management Repository Gap Report

## Current State vs. Discovery Findings

The initial Level 1 implementation of Case Management (ontology and taxonomy YAMLs) must be completely overhauled based on the ADR-023 strict analysis. The current repository likely models too many operational facts as standalone Entities.

### 1. ADR-023 Alignment Failures (The "Entity Explosion")
If the repository models `Referral`, `FollowUp`, and `CaseNote` as independent Entities, it violates ADR-023's principle against promoting fact-bundles to Entities. 
- **Correction Required:** These must be refactored into **Value Objects** embedded within the `CasePlan` or `CaseTimeline`. They have no independent identity outside the context of their parent Case.

### 2. Missing Roles
- **Delegated Authority & Statutory Ownership:** The repository likely lacks the concept of `StatutoryOwner` and `LeadCoordinator`. These must be modeled as **Roles** (per ADR-023) assumed by `Actor` entities, rather than hardcoded entity types.
- **Case Assignment:** The act of assigning a worker should be modeled as a Role assignment, not a standalone `CaseAssignment` entity.

### 3. Missing Runtime / Reasoning Objects
- **Escalations & Recommendations:** The repository must recognize that "Risk of Referral Expiration" or "Suggested Intervention" are **Runtime Objects** generated by the reasoning engine, not permanent domain Entities.
- **Review Decisions:** The outcome of a Case Conference should be a Runtime Object that triggers a state change in the Case Plan.

### 4. Taxonomy Gaps
- The taxonomy must be expanded to handle `SuspensionReason` (e.g., Funding Exhausted, Statutory Hold), which is critical for distinguishing a paused case from a closed one.
- `Graduation` must be explicitly separated from `Administrative Closure` in the outcome taxonomy.
